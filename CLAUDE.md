# CLAUDE.md — نظام إدارة أعمال الرؤية العربية

> هذه الوثيقة هي **مصدر الحقيقة** للمشروع. تُحمَّل تلقائياً لكل جلسة/وكيل. اقرأها أولاً قبل أي عمل — تُغني عن إعادة اكتشاف السياق (توفير ميزانية).

## ١. هوية المشروع
- الشركة: **الرؤية العربية للتجارة العامة — المكتبة العربية للطباعة والقرطاسية**.
- السوق: **العراق** (عربي RTL، عملة دينار عراقي IQD، **بلا ضريبة VAT** افتراضياً = معدّل 0%).
- النشاط: مطبعة + مكتبة قرطاسية، **فرعان** (رئيسي MAIN كل الخدمات + فرع مبيعات SALES). قطاعات: طباعة، قرطاسية (مفرد/جملة)، هدايا/تخرج، تجهيزات مكتبية، عملاء أفراد/تجار/شركات/دوائر حكومية.
- v2 **إعادة تأسيس نظيفة** بعد مشروع Manus (الإرث مؤرشف في `_legacy/`، مُستثنى من git).

## ٢. التقنية والبنية
- React 19 + Vite + Tailwind v4 + shadcn/ui + wouter + TanStack Query؛ tRPC v11؛ Express؛ Drizzle ORM (mysql2) + MySQL 8؛ decimal.js؛ TypeScript.
- مصادقة محلية مملوكة (JWT عبر jose + scrypt) — **لا OAuth ولا Manus**.
- البنية: `client/` (React) ، `server/` (Express+tRPC: `routers.ts`, `routers/*`, `services/*`, `trpc.ts`, `context.ts`, `db.ts`, `auth/`) ، `shared/` ، `drizzle/schema.ts` (٢٨ جدولاً).

## ٣. التشغيل (Windows + PowerShell)
```
docker start erp-mysql            # حاوية MySQL (mysql:8, root/erp_root_pw, منفذ 3306, قاعدة erp)
pnpm install
pnpm db:push                      # هجرة المخطط
pnpm seed                         # admin + فرعان + منتجات عيّنة (idempotent)
pnpm dev                          # http://localhost:3000
pnpm check                        # فحص الأنواع (tsc)
pnpm test                         # اختبارات (vitest) ضد قاعدة erp_test
```
- دخول: `admin@alroya.local` / `Admin@12345` (من `.env`). الإعدادات في `.env` (مُستثنى من git).
- **علل بيئية محسومة:** `tsx watch` معطوب على هذا الجهاز → أمر `dev` يستخدم `node --watch-path=./server ... --import tsx`. Docker Desktop ينهار بعلّة «Inference manager» → عند الحاجة: أوقف كل عمليات docker ثم أعد التشغيل. الكوكي `sameSite:"lax"` ليعمل على localhost.

## ٤. القاعدة الحاكمة (Definition of Done) — لا تُكسَر
> **اكتمال أي وحدة = خلفية (API) + واجهة (شاشة) + فحص (`pnpm check`) + تحقّق (اختبار/جولة بصرية) = ١٠٠٪.**
> الخلفية وحدها بلا شاشة = **وهم اكتمال**. تُبنى كل وحدة **شريحة رأسية كاملة** (زر في الواجهة → خدمة → معاملة ذرية → DB → عرض)، تُتحقَّق بصرياً، وتظهر في **قائمة التنقّل** — ثم ننتقل للتالية. الترتيب: الكاشير أولاً، ثم وحدة-وحدة.

**⛔ بوّابة بدء الجلسة الإلزامية (قبل أي كتابة كود — لا استثناء):** «نفّذ فوراً» أدناه **مشروطٌ** باجتياز هذه البوّابة أولاً. قبل اختيار/بدء أي وحدة:
> ١) **افحص الجلسات المتزامنة:** اسرد جلسات المشروع النشطة (`mcp__ccd_session_mgmt__list_sessions`) + `pnpm coord:list` (الصورة الحيّة للشرائح المُدّعاة)؛ أي جلسة `isRunning` في نفس المجلد = مالكة محتملة — لا تلمس شريحتها.
> ٢) **افحص العمل غير الملتزم:** `git status`؛ أي ملف معدّل/غير متعقَّب = شريحة جلسة أخرى جارية. **ممنوع** البدء بوحدة تتقاطع معه (افحص بدلالة `git diff` لا التخمين).
> ٣) **اعزل جلستك ثم ادّعِ شريحتك:** `pnpm session:new <موضوع>` (ينشئ فرع `session/<موضوع>` + قاعدة/منفذ/‎.env مستقلّين + يسجّلك)، ثم `pnpm coord:claim <موضوع> --files <ملفاتك>` **قبل أي كتابة**. الكتابة على `main` محظورة آلياً بـ hook. إن وجدت WIP لغيرك ⇒ **توقّف وأبلغ** — لا تطمس عمله.
> ٤) عند اكتشاف تقاطع لحظي أثناء العمل (ملف «تغيّر منذ القراءة»، أو ظهور ملفات/تجهيز فهرس لم تصنعه): **توقّف، لا تطمس، أبلغ المالك** — التزامن قرار تنسيقٍ للمالك لا قرارٌ تقنيٌّ تتخذه وحدك.
>
> **سبب القاعدة (٦ يونيو ٢٠٢٦):** ٣ جلسات تسابقت على `main` على شريحة المشتريات نفسها فتضاربت وتكرّر العمل. الفحص المسبق + العزل يمنعان ذلك حتماً.
>
> **🔒 الإنفاذ الآلي (coord) — لم يعد البروتوكول طوعياً:** `scripts/coord-core.mjs` + hooks في `.claude/settings.json` يُنفّذان البوّابة ميكانيكياً: **قفل ذرّي للشريحة** (`O_EXCL` ⇒ سباق مستحيل)، و**رفض الكتابة** على `main`، أو على ملف تملكه جلسة أخرى حيّة، أو على ملف ساخن لغير قائد الدمج (`_integration`). أوامر: `pnpm coord:list|status|claim|release|reclaim`. مخرج طوارئ المالك: `COORD_BYPASS=1`. التفصيل في `docs/playbook.md`.

**أسلوب العمل (تفضيل دائم — مُلزِم):** تصرّف باستقلالية كموظف محترف حريص وأمين ودقيق: **بعد اجتياز بوّابة بدء الجلسة، اختر الخطوة التالية الأصحّ ونفّذها فوراً** بالمنهجية المُحكَمة (بناء → فحص → تحقّق → مراجعة)، وأبلغ بالنتائج لا بالخيارات. **لا تعرض قوائم خيارات ولا تسأل «كيف/أيّهما نبدأ»** إلا عند قرار عملٍ مصيري غير قابل للتراجع (سياسة مالية أو حذف بيانات). الأولوية تتبع §٦ (الحالة) و§٩ (التوصيات).

**دورة كل جلسة (فريق وكلاء ذكي تحكمه القاعدة الحاكمة):** اختر مهمة → أنجزها **١٠٠٪ بكامل تفاصيلها** (خلفية + واجهة + شاشات + كل التحكمات والوظائف الإنتاجية والعملية، بأمان واعتمادية واستقرار) → **تحقّق + فحص + اصطياد الأخطاء وإصلاحها + متابعة** حتى التكامل الكامل → **commit + تقرير موجز ثم إنهاء الجلسة**. استعمل تنظيم الوكلاء (عمّال/مُحقِّقون/مكتشفو نواقص عبر Workflow أو مهارة `review-module`) **عند الحاجة** فقط (§٧) موازنةً مع الميزانية.

## ٥. اتفاقيات الكود (مُلزمة — مُثبَتة بـ٣٢ اختباراً)
- **الذرّية:** كل عملية أعمال داخل `withTx(tx => …)` (server/services/tx.ts)؛ أي throw ⇒ ROLLBACK كامل.
- **المخزون:** بالوحدة الأساس (int) لكل (variant × branch) في `branchStock`. كل تغيير عبر `inventoryService.applyMovement` (يكتب حركة + يحدّث الرصيد تحت قفل `.for("update")`). `baseQuantity = quantity × conversionFactor` ويجب أن يكون صحيحاً.
- **الأموال:** كلها عبر `decimal.js` + `money.ts` (round2 HALF_UP، toDbMoney نصاً). **ممنوع** `parseFloat`/`Number` على الأموال.
- **الدفتر (مبسّط، تلقائي):** قيد عند كل بيع/شراء/دفع/مرتجع. revenue=صافي قبل الضريبة، amount=الإجمالي، RETURN بقيم سالبة. الذمم: `customers.currentBalance` (AR موجب=مدين)، `suppliers.currentBalance` (AP).
- **المنتج:** `products` → `productVariants`(لون/قياس/تكلفة) → `productUnits`(قطعة/درزن/كرتون + معامل + باركود مستقل) → `productPrices`(سعر صريح لكل وحدة×فئة RETAIL/WHOLESALE/GOVERNMENT).
- **idempotency البيع:** `clientRequestId` → `invoices.sourceId` + قيد فريد `(sourceType,sourceId)` + إعادة محاولة الراوتر على `ER_DUP_ENTRY`.
- **الطباعة:** `client/src/lib/printing/` — ESC/POS **raster** (للعربية) عبر WebUSB، وبديل حوار المتصفّح 80مم. قوالب: فاتورة / Z-report / رصيد افتتاحي.

## ٦. الحالة الحالية (يونيو ٢٠٢٦)
**١٠٠٪ (خلفية+واجهة+تحقق):** المصادقة، لوحة التحكم+التنقّل، الكاشير (بيع نقدي)، العمود الفقري (بنية)، الطباعة (بديل المتصفّح؛ WebUSB يحتاج جهازاً للتحقق)، **إضافة منتج وتعديله**، **المشتريات (إنشاء + استلام جزئي/كامل + دفعة مورد)**، **المرتجعات (إرجاع من فاتورة + إعادة للمخزون + استرداد)**، **التحويلات بين الفروع (حركتان ذرّيتان بقفل ثنائي بلا قيد محاسبي)**، **أوامر الشغل/المطبعة (دورة كاملة + فاتورة عند التسليم)**، **تفاصيل الفاتورة + تسديد دفعات للذمم**، **المصروفات اليومية (شاشة + receipt OUT + قيد PAYMENT_OUT + إلغاء يعكس الصندوق)**، **أعمار الذمم المدينة (AR) والدائنة (AP) + كشف حساب عميل ومورد (تقارير قراءة)**، **إدارة العملاء (قائمة + بحث + فلاتر + تقسيم صفحات + إضافة + تعديل + تعطيل/تفعيل مع حماية الرصيد والفواتير غير المسوّاة — ٢٨ اختباراً)**.
**عرض فقط (يلزم شاشات إدخال/تعديل):** المبيعات، حركات المخزون.
**⚠️ فرع معلّق غير مدموج:** `session/pos-credit` — شريحة البيع الآجل/العميل في الكاشير (POS.tsx + CustomerPicker.tsx) مكتملة لكنها التُزمت سهواً على فرع آخر فعُزلت؛ يلزم إعادة `launch.json` لـalroya/3000 قبل دمجها.
**الخطوة التالية المقترحة (شرائح مستقلّة بلا تقاطع):** **دمج/مراجعة فرع `session/pos-credit` المعلّق**، **طباعة ملصقات الباركود** (للأصناف بلا باركود مصنّعي)، **عروض الأسعار** (Quotation → تحويل لفاتورة)، **تشغيل الأدوار/الصلاحيات الفعلية** (إخفاء التكلفة عن الكاشير + توصيل auditLogs).
**قرارات مؤجَّلة للمالك:** الضريبة (افتراضي 0% غير شامل)، التكلفة (آخر تكلفة)، سقف الائتمان (غير مُفعَّل)، الدفع الزائد (مسموح).

## ٧. تنسيق الوكلاء والجلسات (منع التعارض + ذكاء الأعمال)
**نظام الوكلاء (عبر أداة Workflow):**
- **عمّال (workers):** يكتبون الكود. التوازي على ملفات مختلفة فقط، وبعزل **worktree** (`isolation: "worktree"`). **كاتب واحد لكل ملف/وحدة**.
- **مُحقِّقون (verifiers):** مراجعة عدائية متعددة العدسات (صحة/أموال/ذرّية/أمان/UX)، **للقراءة فقط** → يحكمون على كل ملاحظة (تأكيد/رفض). لا يعدّلون ⇒ لا تعارض.
- **مكتشفو النواقص (gap-finders):** ناقد اكتمال يسأل «ما الناقص؟ شاشة؟ تدفّق؟ ادعاء غير مُتحقَّق؟» → يغذّي الجولة التالية.
- النمط القياسي لكل وحدة: تصميم (لوحة خبراء) → تنفيذ (عامل) → تحقّق (عدائي) → كشف نواقص → إصلاح. أمثلة مُطبَّقة: ورشة تصميم العمود الفقري، ومراجعته العدائية التي كشفت ٧ أخطاء.

**🤝 بروتوكول الفريق (التوازي بلا تعارض = عزل + تنسيق + دمج — الطبقات الثلاث):**
> العزل وحده لا يكفي؛ التوازي الآمن يحتاج **قائداً واحداً ينسّق**، لا عدّة جلسات متكافئة تتسابق.
- **١) قائد واحد (orchestrator):** جهةٌ واحدة (جلسة قائدة أو سكربت Workflow) تُحلّل الشريحة، **تعرّف العقود** (شكل الواجهة بين الـAPI والشاشة)، توزّع **وحدات غير متقاطعة**، ثم تدمج. لا وكيل يختار لنفسه عملاً يتقاطع مع غيره.
- **٢) العقد قبل التوازي:** القائد يثبّت شكل المدخلات/المخرجات بين الطبقات **قبل** إطلاق العمّال ⇒ تبني الشاشة على عقدٍ متّفقٍ عليه فيتكامل مع الـAPI بلا انجراف.
- **٣) ملكية ملفات صريحة:** كل وحدة تُعلن ملفاتها — **كاتب واحد لكل ملف**. وحدتان لا تكتبان نفس الملف أبداً (لا حتى في worktrees منفصلة، وإلّا فتعارض دمج).
- **٤) 🔥 سجلّ الملفات الساخنة (يملكها القائد وحده، يعدّلها تسلسلياً بعد فراغ العمّال):** `server/routers.ts`، `client/src/App.tsx` (المسارات)، `client/src/components/AppLayout.tsx` (التنقّل)، `drizzle/schema.ts`، `server/seed.ts`. **الوكلاء/الشرائح لا يلمسونها** — هي مغناطيس تعارض الدمج.
- **٥) الدمج + المزامنة مع الريموت:** شرائح صغيرة، دمج متكرّر إلى `main`؛ القائد يطبّق تعديلات الملفات الساخنة + `pnpm check` + الاختبارات + جولة بصرية قبل الإغلاق، **ثم يدفع فوراً: `git push origin main`** ليبقى الريموت (`origin` = GitHub: ahrrfy/business_management_system) مطابقاً للمحلي. القاعدة: **كل دمج إلى `main` يُتبَع بدفعٍ إلى `origin/main`** (المحلي والريموت يتقدّمان معاً دائماً). الدفع بعد التحقّق فقط — لا يُدفع عملٌ غير مدموج/غير مُختبَر. التفصيل: [[github-remote]] في الذاكرة.

**🔒 الإنفاذ الميكانيكي (نظام coord) — يحوّل البروتوكول أعلاه من نيّةٍ إلى قانون:**
- **قفل ذرّي للشريحة:** `pnpm coord:claim <شريحة> --files <أنماط>` يكتب قفلاً بـ `O_EXCL` في سجلّ مشترك خارج كل worktrees ⇒ كاتبٌ واحد يفوز، **لا سباق ممكن** حتى عند بدءٍ متزامن (يغلق ثغرة ٦/٦). الأوامر: `coord:list|status|release|reclaim|heartbeat`. السجلّ: `~/.claude/coord/<projectKey>/` (أقفال بمهلة ١٥د + استرجاع متحفّظ للمعطوب).
- **hooks تمنع فعلياً** (`.claude/settings.json`): `PreToolUse` يرفض الكتابة على `main`، أو على ملفٍ تملكه جلسة أخرى حيّة، أو على ملف ساخن لغير مالك `_integration`؛ و`SessionStart` يحقن الصورة الحيّة + تحذير main. **يفشلان مفتوحَين** (لا يعطّلان عملاً مشروعاً) ومخرج طوارئ للمالك `COORD_BYPASS=1`.
- **ورشة الشريحة:** مهارة `build-slice` + `‎.claude/workflows/build-slice.js` تشغّل النمط القياسي كاملاً (عقد→كتّاب متوازون→تحقّق عدائي عبر `review-module`→نواقص→إصلاح→دمج) مع **بوّابة DoD آلية** (لا «نجاح» بلا خلفية+واجهة+فحص+اختبار+تنقّل+جولة بصرية). التفصيل التشغيلي في `docs/playbook.md`.

**عزل الجلسات (git) — إلزامي ومُفعَّل آلياً (انظر «بوّابة بدء الجلسة» §٤):**
- كل جلسة على **فرع + شجرة عمل معزولة** عبر `pnpm session:new <اسم>` (ينشئ `session/<اسم>` + مجلّداً مجاوراً بفهرس git خاص + **قاعدة `erp_<اسم>` و`erp_<اسم>_test` ومنفذاً و‎.env مستقلّين** + يسجّل الجلسة في coord) — **الكتابة على `main` محظورة بـ hook**. للعرض: `pnpm session:list`، وللإزالة بعد الدمج: `pnpm session:remove <اسم>`. الدمج إلى `main` بعد بلوغ ١٠٠٪+تحقّق فقط.
- **قبل أي كتابة:** اسرد الجلسات النشطة + اقرأ `git status`. **لا جلستان تُعدّلان نفس الوحدة، ولا جلستان تتشاركان شجرة عمل/فهرس git على نفس الوحدة.** إن وجدت WIP غير ملتزم لغيرك (أو ملفات جهّزها غيرك في الفهرس): **توقّف وأبلغ المالك** — لا تُكمل ولا تطمس ولا تُرجِع عمله.
- **عزل القاعدة تلقائي:** `session:new` يُنشئ `erp_<اسم>` (تطوير) و`erp_<اسم>_test` (اختبار) لكل جلسة ويكتبهما في `.env` ⇒ زال تصادم قاعدة `erp`/`erp_test` المشتركة. (مع Docker متوقّف: تحذير + بديل يدوي.) لكل جلسة منفذها الخاص أيضاً.
- **تغييرات المخطط:** مالك واحد على فرع واحد في كل مرة (تجنّب تعارض الهجرات).
- **تتبّع الملكية عبر سجلّ coord الذرّي** (`coord:claim`) لا بالاتفاق الطوعي — هو مصدر الحقيقة لمن يملك أي شريحة/ملفات. (نظام المهام TaskCreate/`owner` يبقى للتتبّع داخل الجلسة.)

## ٩. توصيات الشريك التشغيلي (مرتّبة بالأولوية — لم يطلبها المالك لكنها مهمة)
**🔴 عاجل قبل إدخال بيانات حقيقية:**
- **النسخ الاحتياطي + خطر فقد البيانات:** حاوية `erp-mysql` بلا **volume** ⇒ `docker rm` يمحو كل البيانات. الحل: إعادة إنشائها بـ named volume (`-v erp_data:/var/lib/mysql`) + **mysqldump يومي مجدول** (دوّار، ونسخة خارج الجهاز).

**🟠 معمارية التشغيل:**
- **قاعدة بيانات مركزية واحدة** للفرعين (الفرع الثاني يتصل بها عبر الشبكة/VPN) بدل قاعدتين تحتاجان مزامنة معقّدة.
- **نشر إنتاجي**: `pnpm build` + تشغيل تلقائي عند الإقلاع (خدمة Windows/مهمة مجدولة)؛ وإعادة النظر في هشاشة Docker Desktop (علّة AI) — ربما MySQL كخدمة Windows أصلب لمتجر واحد.

**🟡 الأمان والصلاحيات:**
- تفعيل **الأدوار فعلياً** (cashier/manager/warehouse) في الواجهة والإجراءات؛ **إخفاء التكلفة عن الكاشير**. توصيل `auditLogs` (من؟ ماذا؟ متى؟).

**🟢 وحدات أعمال مهمة لنشاطك تحديداً:**
- **المطبعة/أوامر الشغل** (جوهر نشاطك، حالياً جداول فقط): تخصيص + **عربون/دفعة مقدمة** + حالة إنتاج + تسليم.
- **الذمم والبيع الآجل** للشركات/الدوائر الحكومية: كشف حساب عميل + **أعمار الذمم (AR aging)** + متابعة تحصيل.
- **المصروفات اليومية** (تؤثر على صندوق الوردية وЗ-report)، **عروض الأسعار**، **طباعة ملصقات الباركود** للأصناف بلا باركود مصنّعي.

**🔵 جودة هندسية:**
- **CI**: hook قبل الالتزام أو GitHub Actions يشغّل `pnpm check` + `pnpm test` تلقائياً (أتمتة البوابة). تسجيل أخطاء الخادم للتشخيص في المتجر.
- **UX الكاشير**: تشغيل بلوحة المفاتيح/الماسح (اختصارات)، و**تقريب نقدي IQD** (لا فئات أصغر من ٢٥٠/٥٠٠).
- **التحقق من الطباعة الحرارية العربية على جهاز حقيقي** (الاختبار الوحيد المعلّق).
