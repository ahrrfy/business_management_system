# دليل النظام التشغيلي (Playbook) — يُقرأ عند الحاجة فقط (ليس تلقائياً)

> توصيات لتمتين الآليات الأربع: اقتصاد الذاكرة، تعارض الوكلاء، تعارض الجلسات، نظام الوكلاء. (CLAUDE.md يبقى رشيقاً ويُحيل هنا.)

## ١. اقتصاد الذاكرة والسياق (تقليل استهلاك الميزانية)
- **`CLAUDE.md` رشيق**: حقائق ثابتة + مؤشرات فقط (يُحمَّل كل turn ⇒ كل سطر يكلّف). التفاصيل العميقة → ملفات `docs/` تُقرأ عند الحاجة.
- **مواصفة لكل وحدة** في `docs/modules/<name>.md` تُقرأ فقط عند العمل عليها — لا تُحمَّل كلها.
- **سجلّ قرارات (ADR)** مختصر append-only (لماذا: بلا VAT، آخر تكلفة، مصادقة محلية، raster للطباعة...) يمنع إعادة التقرير ويوفّر جدالاً.
- **طقس نهاية كل جلسة**: حدّث ذاكرة `status-next` (مصدر «أين نحن/الخطوة التالية») + `git commit`. شغّل مهارة `consolidate-memory` دورياً لدمج/تقليم الذاكرة فتبقى صغيرة ودقيقة.
- **اعمل مُجزّأً** (وحدة واحدة لكل جلسة) فيقلّ السياق المطلوب. استخدم وكلاء **Explore/بحث** ليهضموا الملفات ويعيدوا الخلاصة بدل تحميل ملفات كاملة في السياق الرئيسي.

## ٢. منع تعارض الوكلاء (agent ↔ agent)
- **الكتّاب**: `isolation: "worktree"` (نسخة git مستقلة لكل وكيل) — والتوازي **فقط على نطاقات منفصلة** (مجلد/وحدة لكل وكيل). **كاتب واحد لكل ملف**.
- **الملفات المحورية المشتركة** (`server/routers.ts`, `drizzle/schema.ts`): **مالك واحد** يدمج إضافات الآخرين؛ لا يكتبها وكيلان معاً (أكبر مصدر تعارض دمج).
- **عقد واجهة أولاً**: ثبّت الأنواع/التواقيع في مكان واحد قبل التوازي، فيبني الجميع على عقد ثابت.
- **المُحقِّقون والمكتشفون = للقراءة فقط** (نوع `Explore` أو أدوات مقيّدة بلا Edit/Write) ⇒ مستحيل أن يتعارضوا.
- تتبّع الملكية عبر `TaskCreate`/`TaskUpdate` (`owner`, `blockedBy`).

## ٣. منع تعارض الجلسات (session ↔ session)
- **فرع لكل جلسة** من `main` (`git checkout -b session/<موضوع>`)؛ اسحب main قبل البدء؛ **ادمج بسرعة** (الفروع المعمّرة = تعارضات أسوأ).
- **قاعدة بيانات لكل جلسة** (أو منفذ Docker مختلف) لتفادي تصادم بيانات التطوير.
- **الاختبار**: عيّن `TEST_DATABASE_URL` خاصاً بكل جلسة (`vitest.config` يقرأه) — لأن `erp_test` **تُفرّغ كل تشغيل**؛ جلستان تختبران معاً = إفساد متبادل. ولا تُشغّل `pnpm dev` على نفس المنفذ من جلستين.
- **المخطط**: مالك واحد للهجرات على فرع واحد في كل مرة.

## ٤. نظام الوكلاء (عمّال / مُحقِّقون / مكتشفو نواقص) — أفضل الممارسات
- **فصل الواجبات**: من يبني لا يُصادق على نفسه. المُحقِّق **مستقل** ومُوجَّه لـ**الدحض** (افترض الخطأ افتراضياً) بعدسات **متنوّعة** (صحة، أموال/ذرّية، أمان، UX).
- **تصويت أغلبية** للملاحظات الحرجة (≥٢ من ٣ مُحقِّقين) لتقليل الإيجابيات الكاذبة.
- **مخرجات بنيوية (schemas)** فالنتائج قابلة للاستخدام آلياً (لا نصّ حرّ).
- **مكتشف نواقص بحلقة «حتى الجفاف»**: كرّر حتى جولتان بلا جديد (لا تكتفِ بمرور واحد).
- **بوابة الدمج**: لا تُدمج وحدة إلا بعد أن يُرجع workflow التحقق **صفر أخطاء حرجة + صفر شاشات ناقصة** (= القاعدة الحاكمة: خلفية+واجهة+تحقق).
- **حجم الفريق حسب الخطر/الميزانية**: كود مالي/مخزون ⇒ مُحقِّقون أكثر؛ شاشة عرض بسيطة ⇒ أقل (وازِن مع الميزانية).
- **Workflows محفوظة** في `.claude/workflows/`: `review-module` (موجود — مُحقِّقون+مكتشفو نواقص). يُنصح بإضافة `design-module` و`build-verify` لاحقاً.
- مثال مُطبَّق نجح: ورشة تصميم العمود الفقري (٤ عدسات) + مراجعته العدائية (٤ عدسات + محقّق) كشفت **٧ أخطاء مالية حقيقية** فأُصلحت + اختبارات انحدار.

## ٥. قواعد العربية/العراق (يلتزمها كل وكيل في كل مخرَج)
- **كل واجهة عربية RTL** — لا تُشحن واجهة إنجليزية. العملة **IQD**، **بلا VAT** افتراضياً، تقريب نقدي للفئات المتاحة (لا أصغر من ٢٥٠/٥٠٠).
- **مصطلحات موحّدة**: درزن (١٢)، كرتون، عربون (دفعة مقدمة)، ذمم مدينة/دائنة (AR/AP)، وردية، تقرير Z، درع، تخصيص، أمر شغل.
