# بنود متابعة تدقيق السلامة المالية — ٢ يوليو ٢٠٢٦

> **الغرض:** هذا هو **دفتر العمل الدائم** للبنود المتبقّية من حملة تدقيق السلامة المالية/المحاسبية/النقدية
> (PR #104، مدموج في `main` عند `74d66ba`). البيئة السحابية للجلسة **مؤقّتة** (تُمحى عند الإغلاق)، فحُفظت
> البنود هنا في المستودع ليُعالَجها المالك **في البيئة التشغيلية التي بها قاعدة بيانات فعّالة**.
>
> **طريقة العمل لكل بند:** نفّذ في بيئة DB → افحص/اختبر → تحقّق → **شطب المربّع** → انتقل للتالي.
>
> **خلفية:** التمريرة الأولى عالجت **٨٩ ملاحظة مؤكَّدة** (أغلبها مُصلَح + مُتحقَّق على ١٢٠٧ اختبار في CI + مدموج).
> البنود أدناه هي ما أُجِّل **بقرارٍ واعٍ** (خطر إنتاج، أو حاجة لقاعدة بيانات للتحقّق، أو قرار سياسة) — لا نسياناً.

---

## لماذا أُجِّلت (لا تُحذَف هذه الفقرة)
لم تُشحَن هذه البنود في التمريرة الأولى لأنّها إمّا: (أ) تُعدّل جدولاً إنتاجياً حيّاً وقد تُفشِل النشر (FK)،
أو (ب) تحتاج قاعدة بيانات فعلية للتحقّق من صحّتها قبل الدمج (رياضة الدفتر)، أو (ج) قرار سياسة/معماري أوسع.
البيئة السحابية للتمريرة الأولى بلا MySQL محلي، فالتحقّق كان عبر CI فقط — وهذه البنود تحتاج تفاعلاً مع DB حيّة.

## F1 — هجرة مفاتيح أجنبية مفقودة  ‏`[x]` ✅ (٢/٧/٢٦)
> **أُنجِز:** أُضيف `.references()` لـ`purchaseOrderId`→`purchaseOrders.id` و`exchangeHouseId`→`exchangeHouses.id` في `schema.ts` +
> هجرة يدوية idempotent ذاتيّة الشفاء `drizzle/migrations/0040_accounting_entries_fk.sql` (تُصفّر الأيتام أولاً ثم تُضيف القيد
> بفحص `information_schema` — لا db:generate لأن snapshot مُجمَّد؛ نمط 0039) + قيد journal 40. **تحقّق فعليّ على DB:** مسار CI
> (`db:push`) ومسار الإنتاج (migrator كامل ٤١ هجرة) كلاهما يُنشئ القيدين بأسماء اصطلاح drizzle؛ إدراج مرجع وهميّ يُرفَض (1452)؛
> re-apply لا يُكرّر (idempotent)؛ تصفير الأيتام يعمل؛ `pnpm check`+`db:verify` أخضر؛ اختبارات الصيرفة/المشتريات ٣٧/٣٧.
- **الخطورة:** منخفض · **المصدر:** ملاحظة #18 · **الملف:** `drizzle/schema.ts` (`accountingEntries`)
- **المشكلة:** `accountingEntries.exchangeHouseId` و`accountingEntries.purchaseOrderId` بلا `FOREIGN KEY`
  (تعليق المخطط يدّعي أن الـFK «يُضاف في الهجرة» ولم يُضَف). لا حارس تكامل مرجعيّ.
- **لماذا أُجِّل:** إضافة FK على جدول إنتاجيّ حيّ **تفشل** إن وُجد صفٌّ يتيم ⇒ توقُّف النشر.
- **الخطة (في بيئة DB):**
  1. افحص الأيتام أولاً:
     ```sql
     SELECT COUNT(*) FROM accountingEntries ae LEFT JOIN exchangeHouses x ON ae.exchangeHouseId=x.id
       WHERE ae.exchangeHouseId IS NOT NULL AND x.id IS NULL;
     SELECT COUNT(*) FROM accountingEntries ae LEFT JOIN purchaseOrders po ON ae.purchaseOrderId=po.id
       WHERE ae.purchaseOrderId IS NOT NULL AND po.id IS NULL;
     ```
  2. إن وُجد أيتام: صفّرها (`SET ...Id = NULL WHERE ...`) — قرار مالك (هي بيانات خطأ).
  3. أضِف `.references(() => exchangeHouses.id)` و`.references(() => purchaseOrders.id)` في `schema.ts`.
  4. `pnpm db:generate` (محلياً) → راجع SQL المُولَّد → `pnpm db:migrate:safe` على الإنتاج.
- **التحقّق:** `pnpm db:verify` أخضر + صفر أيتام + محاولة إدراج قيدٍ بمعرّفٍ وهميّ تُرفَض.
- **DoD:** الـFK موجودان في `information_schema.KEY_COLUMN_USAGE` + النشر لم يتعطّل.

## F6 — اختبار توصيف لعكس تقريب IQD عند المرتجع الكامل  ‏`[x]` ✅ (٢/٧/٢٦)

> واحدة (ADJUST إنشاء −adj + ADJUST مرتجع +adj) ⇒ صفر بقايا تقريب في P&L، لا عكس مزدوج. حالتان (مقرَّب ١٣٠٠⇒١٢٥٠
> بتقريب −٥٠؛ وcontrol غير مقرَّب بلا ADJUST) خضراوان + تعادُل النقد (المستلَم=المُسترَدّ). توصيف لا إصلاح (السلوك سليم).
- **الخطورة:** متوسط (تحقّق لا إصلاح) · **المصدر:** ملاحظتا #5 و#26 · **الملف:** `server/services/returnService.ts`
- **الحالة:** التحليل يُظهر أن عمود `amount` **يتصافر** في الحالة العراقية الافتراضية (بلا ضريبة): بيع=`effectiveTotal`،
  مرتجع=`−effectiveTotal`، و`returnedTotal` من أسطرٍ قبل التقريب ⇒ لا عكس مزدوج. لم يُضَف اختبار مخصّص.

  (إن فُعِّلت) لكشف أي اختلال في `amount` غير المتعلّق بالتقريب.
- **التحقّق:** الاختبار أخضر ⇒ يوثّق الصحّة؛ إن سقط ⇒ يكشف الرقم الفعليّ ويُصلَح.
