> **الحالة:** تصميم مُحقَّق للتنفيذ · **لا كود بعد وقت كتابة الوثيقة** (بدأ التنفيذ بالشريحة ٠).

## ٠) الهدف والقرارات المقفولة

**الفكرة:** عند رفع صورة منتج، يُنشئ النظام **نسخة استوديو مقترحة** (خلفية بيضاء نظيفة + قالب موحّد + ظلّ)، يراجعها موظفٌ ويعتمدها **قبل** ظهورها في المتجر. الأصل لا يُستبدَل ولا يُنشر مباشرةً.

**قرارات المالك المقفولة (§١١ من الحوار):**
1. **سقف ميزانية مسار Pro عند المزوّد** نفسه (لا يقيس التطبيق الإنفاق — يتدهور بسلاسة عند التجاوز).
2. **أمانة صارمة: يُمنَع منعاً باتاً توليد/تعديل بكسلات المنتج** — الخلفية فقط. (منتجات كثيرة عليها نصّ عربيّ أو ملفوفة سلوفان/شفافة/عاكسة.)
3. **خلفية بيضاء `#FFFFFF` موحّدة** بنمط عالميّ احترافيّ خفيف (لا مشاهد مولَّدة).
4. **تخزين هجين آمن يتوسّع** مع نمو الكتالوج.

**عقد الجودة المُلزِم:** شفافية عالية · جودة عالية · اعتمادية واستقرار · أمن · تنظيم عالٍ · بناء متين · **تكلفة تشغيل ~صفر** (مدفوع اختياريّ محكوم).

## ١) القرار المعماري لبنية التخزين الهجينة

**التوصية الحاسمة الواحدة:** مخزن كائنات خارجي S3-متوافق = **Cloudflare R2**، **معنون-بالمحتوى** (المفتاح = هاش البايتات)، خلف واجهة **`ImageStore`** قابلة للاستبدال. **لا نظام ملفات محلّي على الـVPS لكامل الدقة.**

`productImages` يصير **سجلّ مؤشّر + معاينة منخفضة**: المصغّرة (`thumbDataUrl`) + البصمة (`contentHash`) + الوصف يبقون في MySQL؛ الأصل والمعالَجة كاملة الدقة في R2.

**المبرّر:**
1. **الخطّ الأحمر يحسم وحده:** الـVPS مشترك مع سراج/أودو و«لا يُملأ القرص». أصولٌ كاملة الدقة لكتالوجٍ ينمو من ١٤٥٣+ = نموّ قرصٍ غير محدود ⇒ **نظام الملفات المحلّي مرفوض بنيوياً**.
2. **Cloudflare أمام النطاق أصلاً** (Full strict على `alarabiya.online`) ⇒ R2 + قاعدة كاش حافة + `?v=<hash>` = كاش أبديّ يتوسّع مع الكتالوج بفعل مالكٍ واحد في لوحة CF.
3. **صفر egress في R2** — الصور تُخدَم لكل زائر دون فاتورة نطاق متصاعدة.
4. **التعنون-بالمحتوى** يحوّل الكتابة/النسخ المزدوج إلى مسألةٍ تافهة: الكائن ثابتٌ ببنائه، PUT متعادِل، «الكائن أولاً» يمنع المرجع المعلَّق حتماً، والمرآة تراكميّة رخيصة.

الواجهة (`put/head/getStream/signedUrl/delete`) تُبقي مخرجاً (MinIO/B2/S3) وسائق **نظام-ملفات لبيئات الجلسة/التطوير حصراً** (منفذ 3310، لا قرص مشترك).

### نموذج البيانات — هجرة `0095`

SQL يدويّ + مُدخَل `_journal.json` (MySQL 8 بلا `ADD COLUMN IF NOT EXISTS`؛ جُمَل `--> statement-breakpoint`). أعمدة **NULLable** على `productImages`:

| العمود | النوع | الدور |
|---|---|---|
| `objectKey` | VARCHAR(255) | مفتاح R2 للمعالَجة = `<companyPrefix>/p/<h0h1>/<fullhash>.<ext>`. NULL=صفّ إرثيّ على `url`. |
| `originalKey` | VARCHAR(255) | مفتاح R2 **للأصل غير الممسوس** (عكوسيّة قرار ٢). ثابتٌ عبر إعادات المعالجة. |
| `contentHash` | CHAR(64) | **sha256 لبايتات المعالَجة المفكوكة** — البصمة القانونية الوحيدة (مصدر `v=`/ETag/بصمة الأوفلاين). |
| `thumbDataUrl` | MEDIUMTEXT | المصغّرة (~٦٤px، ١-٣ ك.ب) — تبقى في DB، شبكة أمان العرض. |
| `mime`,`width`,`height`,`bytes` | — | مثبَّتة وقت الرفع (تُخدَم لا تُستنشَق). |
| `reviewStatus` | ENUM('APPROVED','PENDING_REVIEW','REJECTED') default `APPROVED` | بوّابة النشر (توافق خلفيّ). |
| `origin` | ENUM('ORIGINAL','STUDIO_FREE','STUDIO_PRO','MANUAL') default `ORIGINAL` | حوكمة/عكس. |
| `migratedAt` | TIMESTAMP NULL | مؤشّر استئناف الترحيل. |

**قاعدتا هاش مُوحَّدتان صراحةً** (تُصلحان ضعف اليوم — `imageHash` يهشّ نصّ data-URL لا البكسلات): `contentHash` + مفتاح الكائن = sha256 كامل على **البايتات المفكوكة**؛ `?v=` = أوّل ١٦ hex من نفسه، يُقرأ من العمود لا يُعاد حسابه.

## ٢) الأمان والاعتمادية والاستقرار

**أ. الذرّية — «الكائن أولاً، ثم الصفّ»:**
1. متصفّح الأدمن يُنتج المعالَجة + المصغّرة ويحسب `contentHash`.
2. الخادم يتحقّق (§٥ الحواجز) **ثم يُودِع الكائن في R2 عند `key=contentHash` أوّلاً** — PUT متعادِل (كائن موجود = HEAD يؤكّد، لا كتابة فوقية).
3. **بعد تأكيد 2xx فقط** يُفتَح `withTx` ويُكتَب صفّ البيانات الوصفية ⇒ المرجع المعلَّق **مستحيلٌ بنيوياً**.
4. **كل الـPUTs خارج وقبل `withTx`** ⇒ لا معاملة DB مفتوحة عبر رفع شبكيّ.

**ب. استقرار المفتاح — تحديث في المكان لا حذف+إدراج [CONFIRMED]:** `productEditService.ts:408-410` يحذف-ثم-يُدرج صورة المتغيّر ⇒ **id جديد كل حفظ** ⇒ يكسر الرابط المُفتَّح بـ`productImages.id`+immutable+ETag بصمت، ويُدلّي كاش SW (سنة)/حافة CF/صفوف الأوفلاين. **القاعدة الإلزاميّة:** مسار الاستبدال/إعادة المعالجة **يُحدِّث الصفّ القائم في مكانه** (`UPDATE … WHERE id=`)، لا حذف+إدراج. **إصلاح مسار المتغيّر القائم شرطٌ مسبقٌ للترحيل** (علّة قائمة اليوم قبل R2 أصلاً).

**ج. اليتامى والتباعد:**
- **يتيم أماميّ** (كائن بلا صفّ): حميد. **كنس مؤجَّل معدود-مرجعياً**: عاملٌ دوريّ يطرح مفاتيح R2 من `SELECT objectKey UNION SELECT originalKey FROM productImages` ويحذف فقط اليتيم ∧ الأقدم من الأفق (يُبطل خطر حذف كائنٍ مشترك بين منتجين متطابقي البايتات).
- **مرجع معلَّق**: يمنعه ترتيب (أ). تسويةٌ دوريّة (نظير `db:verify`) تبثّ `objectKey` وتـHEAD ⇒ **إنذار لا حذف تلقائي** + استعادة من versioning.
- **🔒 قاعدة الأفق مُرمَّزة ومفحوصة لا نصّية [CONFIRMED]:** أفق الكنس/lifecycle **≥ max(احتفاظ النسخ ~٩٠ يوماً، أقصى تقادم جهاز أوفلاين)**. (١) **لا حذف صلب** — versioning + lifecycle بتحويلٍ لطبقة أرخص فقط. (٢) **مسبار بنمط `db:verify`** يأخذ عيّنة `objectKey` من أقدم نسخة قاعدة محتفَظ بها ويـHEAD عليها ⇒ يفشل بصوت عالٍ عند أي غياب. (٣) يُوثَّق في DR كثابتٍ يُختبَر.

**د. ثابت المصغّرة الذرّي [PLAUSIBLE]:** اكتب `objectKey` و`thumbDataUrl` **معاً في نفس الـUPDATE الذرّي**؛ فحص `db:verify` يرفض `objectKey IS NOT NULL AND thumbDataUrl IS NULL`. توليد المصغّرة **شرط قبول الرفع** لا خطوة لاحقة.

**هـ. التحقّق من `?v=` عند الأصل [CONFIRMED]:** المسار الحاليّ يخدم البايتات متجاهلاً `?v=`. **الإصلاح:** قارن `?v=` بـ`row.contentHash.startsWith(v)` وأعِد 404 عند عدم التطابق ⇒ نسخةٌ واحدةٌ قابلةٌ للتخبئة لكل محتوى، يستحيل إغراق مفاتيح CF أو تسميم كاش.

**و. الخدمة تبقى بروكسي مُبوَّباً، لا redirect عام [CONFIRMED]:**
- `/api/img/product/:id` يبقى **بروكسي/بثّ** (`imageStore.getStream(objectKey)` أو `decodeDataUrl(url)` للإرث) **مع إبقاء `sendImage()` حرفياً**: بوّابة `isActive && !isService && showInStore` (العلنيّ) / مصادقة + `private` + `Vary:Cookie` (الكشك) + MIME whitelist + nosniff + المفتاح `productImages.id`.
- **خطّ أحمر:** bucket R2 **خاصّ** — علنيّ القراءة يسرّب صور `showInStore=false` والكشك لأي مخمّن هاش.
- **تبعية شبكة خارجية [CONFIRMED]:** اخدم **المصغّرة من DB مصدراً أوّل** وحُدّ R2 للأصل كامل-الدقة والتنزيل الصريح؛ أضِف **مهلة + دائرة قطع** على `getStream` مع سقوطٍ فوريّ للمصغّرة؛ **اجعل معدّل إصابة كاش CF بوّابةَ نشر** قبل أي قصّ.
- **تسميم كاش المصغّرة الاحتياطيّة [CONFIRMED]:** ردّ المصغّرة عند إخفاق R2 يحمل **`Cache-Control: no-store`** (لا immutable أبداً) ويمرّ فرعاً منفصلاً؛ غياب الكائن = خطأ قابل لإعادة المحاولة لا يُخبَّأ.

**ز. أساسا الهاش أثناء الانتقال [PLAUSIBLE]:** ثبّت تعريفاً قانونياً واحداً (`contentHash` = بايتات مفكوكة) مصدراً وحيداً لـ`v=`/ETag/بصمة الأوفلاين؛ اقلب `v=` **مرّةً واحدةً بالضبط** لكل صورة عند الترحيل، مجمَّعاً مع رفع بادئة الأوفلاين v2→v3.

**ح. النسخ المزدوج والاستعادة المتّسقة:**
- **نصف القاعدة** (وصف + مصغّرات): الآليّة بلا تغيير، لكنه **يتقلّص جذرياً**.
- **نصف المخزن** (كامل الدقة): R2 (منسوخ لدى المزوّد) + versioning/lifecycle + **مرآة باردة مستقلّة**: وسّع سحب المتجر اليوميّ ليضمّ **`rclone sync`** للـbucket ⇒ جهاز المتجر يمسك معاً `.sql.gpg` + مرآة كائنات كاملة.
- **escrow (DR §٤.١):** أضِف حساب R2 + access/secret key + bucket + endpoint؛ السرّ وقت التشغيل عبر `integrationService` + `INTEGRATIONS_ENCRYPTION_KEY` (AES-256-GCM).
- **اختبار DR الربع-سنويّ يتحقّق من الاثنين:** استعادة `.sql` **و** حلّ صورة عيّنة من R2/المرآة.

## ٣) أثر الأوفلاين

**الحقيقة (مؤسَّسة بالقراءة):** الصور **خارج لقطة الأوفلاين تماماً اليوم** (`buildCatalogSnapshot` لا يلمس `productImages`؛ الكاشير نصّيّ بحت). القرار: **مرجع نحيل لا بايتات.**

- `OfflineCatalogRow` يكتسب حقلين نصّيّين صغيرين فقط: `imageId: number|null` + `imageHash: string|null` (≈٢٥ بايت/صفّ). **صفر بايت صورة في اللقطة.** الكامل يُجلَب أونلاين عبر `<img src="/api/img/product/{imageId}?v={imageHash}">` وكاش الـSW القائم يوفّر «أوفلاين-بعد-أول-عرض».
- **رفعتان إلزاميّتان في شحنة واحدة:** (أ) بادئة الاشتقاق **v2→v3** في `catalogVersionParts`؛ (ب) **بصمة CRC سادسة على `productImages`** تعتمد **عمود `contentHash` المخزَّن** لا `crc32(url)` على MEDIUMTEXT (كارثة أداء كل ١٥د). بلا هذه البصمة، استبدال صورة لا يقلب `catalogVersion` فتبقى قديمة للأبد.
- **مصغّرات كاشير «باردة»** (إن طلبها المالك): جدول Dexie منفصل `thumbs` مفكوك عن `catalogVersion` — لا حشو في صفوف الكتالوج المتكرّرة. الأرجح: مرجع-فقط + كاش SW يكفي.
- لا تبتلع أخطاء `syncOfflineCache` صامتاً ⇒ إنذار «فشل مزامنة مستمرّ».

## ٤) خطة ترحيل الـ١٤٥٣ بلا توقّف

النمط المرجعيّ = **حملة تنظيف الأسماء** (قفل تفاؤليّ + دفعات + استئناف)، **لا** `importService` (معاملة كلٌّ-أو-لا-شيء).

**عامل الترحيل** (idempotent، مقفول تفاؤلياً، مُدفَّع؛ سابقة `erp-provision-worker`، خارج مسار الطلب وخارج الذروة):
```
SELECT id,url FROM productImages WHERE migratedAt IS NULL AND url LIKE 'data:%' ORDER BY id LIMIT :batch  -- 25-50
لكل صفّ: decode → contentHash(bytes) → thumb(64px) → PUT R2 (اكتب-إن-غاب)
UPDATE productImages SET objectKey=,contentHash=,thumbDataUrl=,migratedAt=NOW()
 WHERE id=:id AND migratedAt IS NULL AND url=:urlSnapshot   -- قفل تفاؤليّ: تعديل متزامن ⇒ 0 صفوف ⇒ يُلتقَط لاحقاً
```

**تناقض حرِج محسوم [CONFIRMED]:** الـ١٤٥٣ محبوسة base64 ولا يمكن معالجتها في متصفّح ⇒ العامل يفكّ/يصغّر خادميّاً، وهو ما حظره MVP على VPS مشترك. **الإصلاح (بالترتيب):** (أ) **ولّد مصغّرات الإرث عميليّاً** عبر أداة دفعة إداريّة في المتصفّح؛ أو (ب) اعزل مكتبة أصليّة في عملية عامل منفصلة (خارج الذروة، دفعات صغيرة، حدّ ذاكرة صريح)؛ أو (ج) أجّل المصغّرة (عنصر نائب + اشتقاق كسول عميليّ).

**قِس أوّلاً** `SELECT COUNT(*), SUM(LENGTH(url)) FROM productImages` وتأكّد هامش القرص.

### ترتيب العمليات الآمن
0. **بنية (مالك):** R2 خاصّ + اعتماد في `integrationService` + versioning + أفق lifecycle ≥ احتفاظ DB + escrow.
1. **هجرة 0095** (أعمدة NULL؛ `url` بلا مسّ). صفر توقّف. القراءة ١٠٠٪ من `url`.
2. **مسار الاستوديو + القراءة المزدوجة** (`objectKey ? store.get : decodeDataUrl(url)`)؛ الصور الجديدة مُرحَّلةً؛ **إصلاح تحديث-في-المكان (§٢.ب)**.
3. **شغّل عامل الترحيل** خارج الذروة حتى المؤشّر ١٠٠٪.
4. **الأوفلاين:** imageId/imageHash + بادئة v2→v3 + بصمة سادسة — **شحنة واحدة بعد اكتمال الترحيل**.
5. **نقع (soak):** عدّاد إصابات فرع `url` = صفر + مطابقة كل `objectKey` + دورة نسخ كاملة + versioning حيّ + معدّل إصابة كاش CF عالٍ. **حدّث `docs/disaster-recovery.md §٥`** (استدلال «تقلّص النسخة = فقد» إنذار كاذب وقت القصّ).
6. **[LOW]** `ALTER TABLE productImages MODIFY url MEDIUMTEXT NULL` **قبل** القصّ (العمود `NOT NULL` اليوم).
7. **هجرة القصّ:** `UPDATE … SET url=NULL WHERE migratedAt IS NOT NULL AND objectKey يُحَلّ` — **مسيَّجة آلياً بثلاثة شروط [CRITICAL]:** مرآة rclone مبنيّة ومجدولة ومُتحقَّق أنها كاملة + escrow R2 مضاف + اختبار DR يحلّ صورة فعلاً. أبقِ العمود + فرع البديل **دورة نسخ إضافية**.
8. **لاحقاً (ثقة عالية):** `DROP COLUMN url` + إزالة فرع البديل.

**العكوسية:** حتى الخطوة ٥ التراجع = تجاهُل المخزن والقراءة من `url`. الخطوتان ٧-٨ فقط غير عكوسيتين ومسيَّجتان.

## ٥) خط أنابيب الاستوديو النهائي

**المبدأ الحاكم:** الأصل مقدَّس، الخلفية وحدها متغيّرة، العلنيّ محكوم ببوّابة بشرية.

**ترتيب العمليات القانونيّ (عميليّ بالكامل في المسار المجاني — يُدرَج بين القراءة والضغط):**
```
raw file (بألفاه الأصلية)   ← لا يُبنى فوق ناتج compressImageDataUrl المُسطَّح على أبيض مسبقاً
 → [1] تحليل ثقة (إحصاء ألفا/حواف) ⇒ confidence + flags + اختيار الوضع
 → [2] CUT (قناع ML @imgly في Web Worker → قصّ) | FLATTEN (بلا قصّ — الآمن)
 → [3] تركيب على القالب البيتيّ الحتميّ (shared/imageStudio/template.ts: canvas مربّع، منطقة آمنة ~82%، #FFFFFF)
 → [4] ظلّ تماس حتميّ (تدرّج إهليلجيّ، معاملات ثابتة)
 → [5] تطبيع أبيض محيطيّ (خارج صندوق القناع حصراً — لا يمسّ الداخل)
 → [6] encodeSmallest ≤700KB
```

**الوضعان (كلاهما يمسّ المحيط فقط؛ عند الشكّ ⇒ FLATTEN):** CUT (ثقة عالية ولا نزيف حافّة) / **FLATTEN الافتراضي الآمن** (الأصل موسَّطاً على أبيض + ظلّ من صندوق الإحاطة، **لا إزالة/تعديل بكسلة منتج**). أسوأ حالة = أبيض نظيف بلا قصّ ⇒ **الانحدار مستحيل أن يأكل منتجاً**.

**مسار Pro** (اختياريّ، صور بطلة): تجريد `ImageStudioProvider` (`removeBackground(png)→cutoutPng`)، السرّ عبر `integrationService`، **معطَّل افتراضياً**. المزوّد يُعيد **القناع فقط**، والتركيب عميليّ مشترك مع المجاني. 402/429 ⇒ **تدهور سلس للمجاني/FLATTEN**.

**حواجز الأمانة الصارمة (كلها `CONFIRMED`):**
1. **[CRITICAL] تجاوز بوّابة الاعتماد عبر id enumeration:** **لا تُجسِّد المرشّح المعلَّق كصفّ `productImages` قابل للخدمة** — احتجزه في `productImageJobs.processedUrl` حتى الاعتماد. + `reviewStatus='APPROVED'` في استعلامَي `/api/img/product/:id` و`/kiosk-product/:id` (البوّابة أمام البايتات لا في إسقاط القائمة وحده).
2. **[HIGH] «لا بكسلة منتج» غير قابل للإنفاذ في CUT:** عرّف «بكسلة المنتج» = الداخل المعتم (alpha=255)؛ كل بكسلة حدّية = مُعدَّلة حتماً. **مراجعة قبل/بعد بشرية إلزاميّة لصور المتجر**؛ `softEdgeRatio` مرتفع (سلوفان/عاكس/شفّاف) ⇒ **إجبار FLATTEN**.
3. **[HIGH] محو محتوى داخليّ (نصّ عربيّ على قرطاسية فاتحة):** **كاشف ثقوب داخلية** + مقارنة مدرّج لمعان + **إجبار FLATTEN + مراجعة إلزاميّة لصنف القرطاسية** + احصر التطبيع خارج صندوق القناع.
4. **[HIGH] قنبلة بكسلات/فكّ ضغط:** `assertValidImageDataUrl` لا يفحص الأبعاد ⇒ **حلّل ترويسة الصورة خادميّاً** (PNG IHDR/JPEG SOFn/WebP VP8) وارفض >٤٠٩٦×٤٠٩٦، على **كل** مسارات كتابة `productImages`.
5. **[HIGH] strictMagic + إعادة ترميز على كل الكتّاب:** `createProduct`/`updateProductVariants` ينادون المُتحقّق بلا `strictMagic` ⇒ افرضه على **كل** كاتب + إعادة ترميز خادميّة.
6. **[PLAUSIBLE] تجريد الميتاداتا:** جرّد EXIF/GPS/ICC على الأصل المخزَّن وأي إرسال Pro (تجريد ميتاداتا ≠ تعديل بكسلات ⇒ يوافق قرار ٢).
7. **[PLAUSIBLE] صدى خطأ المزوّد:** بروكسي فقط + **لا تُظهِر جسم ردّ المزوّد الخام** + امسح `Authorization` من السجلّ.
8. **[PLAUSIBLE] رابط خارجي passthrough:** الاستوديو **يجلب البايتات ويخزّنها دائماً** (لا يُبقي رابط طرف ثالث)؛ قائمة سماح SSRF على مضيف المزوّد.

**بنية اللا-تزامن (صفر استدلال على VPS):** المسار المجاني ML عميليّ ⇒ لا-متزامن أصلاً بلا خادم. العامل `erp-image-worker` (PM2 single-shot، env معزول، **شبكة + DB فقط، صفر canvas/ML خادميّ في MVP**) لـ: آلة حالة المراجعة + دفعات Pro + التدهور بالميزانية.

**بنيتان تشغيليّتان [CONFIRMED]:**
- **نموذج ML (~٤٠م.ب+):** استضِف ذاتيّاً (CSP يحجب CDN)، **استبعده من glob الـprecache** (`vite.config.ts` سقف 5م.ب)، أضِف قاعدة `runtimeCaching` CacheFirst مخصّصة، حمّله كسولاً، اكشف القدرات WebGPU→WASM→تعطيل CUT.
