# تقرير المالك — استيراد بيانات النظام القديم (٢٠٢٦-٠٦-١٠)

> هذا هو الدليل الشامل لاستيراد ملفات تصدير النظام القديم إلى نظام الرؤية العربية الجديد:
> ما الذي يُستورد وما الذي لا يُستورد ولماذا، خريطة كل عمود في ملفاتك إلى حقله في النظام،
> القرارات المالية الحسّاسة (الإشارة، العملة، الرصيد الافتتاحي) بأمثلة محسوبة من بياناتك الحقيقية،
> وخطوات الاستيراد عملياً شاشةً شاشة. مرجعه التقني: `docs/import-integration-contract.md`.

## ١. الملفات الأحد عشر — ماذا يُستورد وماذا لا

| # | الملف | الصفوف | القرار |
|---|---|---|---|
| ١ | مراجعة الأصناف | ٩٬٤١٥ | ✅ **يُستورد** (شاشة المنتجات) |
| ٢ | مراجعة الموردين | ١٨٦ | ✅ **يُستورد** (شاشة الموردين) |
| ٣ | مراجعة العملاء | ٣٢٦ | ✅ **يُستورد** (شاشة العملاء) |
| ٤ | مراجعة مبيعات | ٢٠٤٬٣٤٦ | ⛔ لا يُستورد (انظر §٥؛ مصدر اختياري لاستخراج عملاء **بعد تنظيف** — §٨) |
| ٥ | مراجعة مشتريات | ٢٬٨٥٦ | ⛔ لا يُستورد (§٥) |
| ٦ | مراجعة مرتجع مبيعات | ١٬٠٥٥ | ⛔ لا يُستورد (§٥) |
| ٧ | مراجعة مرتجع مشتريات | ٩١ | ⛔ لا يُستورد (§٥) |
| ٨ | مراجعة استلام نقدية | ٢٬٥٠٦ | ⛔ لا يُستورد (§٥) |
| ٩ | مراجعة دفع نقدية | ٥٤٧ | ⛔ لا يُستورد (§٥) |
| ١٠ | مراجعة الإيرادات | ١٬٠٧٤ | ⛔ لا يُستورد (توصية وحدة مستقبلية لإيرادات المطبعة — §٩) |
| ١١ | مراجعة المصروفات | ٣٬٤٨٤ | ⛔ لا يُستورد (المصروفات اليومية تُسجَّل في النظام الجديد من يوم التشغيل) |

**ترتيب الاستيراد الإلزامي: الأصناف ← الموردون ← العملاء.**
- **الأصناف أولاً:** أكبر ملف وأبسطه مالياً (لا إشارات ولا عملات) — يبني الكاتالوج والمخزون الافتتاحي على قاعدة نظيفة ويُتحقَّق من عدّاداته بسهولة.
- **الموردون ثانياً:** ملف صغير (١٨٦ صفاً) تُمارَس عليه قرارات الرصيد الافتتاحي (اتجاه الإشارة + سعر الصرف) قبل ملف العملاء.
- **العملاء ثالثاً:** بعد أن صار سعر الصرف وقرار الإشارة مفهومَين عملياً.

> الاستيراد يتطلب صلاحية **مدير** (manager). إعادة تشغيل أي ملف **آمنة دائماً**: الموجود يُتخطّى
> (مطابقة بالرقم القديم ثم الهاتف ثم الاسم)، والأرصدة الافتتاحية **لا تتكرّر بنيوياً** (§٤).

## ٢. خرائط الأعمدة — عمود ملفّك ← حقل النظام

الربط **تلقائي**: ترويسات ملفاتك الحقيقية مسجَّلة حرفياً في النظام (بما فيها «حد الإئتمان» بهمزة الملف
و«اخر تعامل» بلا مدّ — توحيد الألف مفعَّل). راجِع شاشة المطابقة قبل التنفيذ دائماً.

### ٢.١ مراجعة العملاء (٣٢٦ صفاً — يُستورد)

| عمود الملف | حقل النظام | ملاحظات |
|---|---|---|
| الاسم | الاسم (إلزامي) | بعض الأسماء أرقام/باركودات (مثل «8437020176569») — تُستورد كما هي؛ نظّفها لاحقاً من شاشة تعديل العميل |
| الرقم | **الرقم القديم** (legacyCode) | المعرّف الطبيعي للملف (٣٢٥/٣٢٥ فريد) — مفتاح المطابقة الأول ومانع التكرار؛ يظهر عموداً في شاشة العملاء |
| تليفون 1 | الهاتف | مملوء ٥٧٪ فقط (١٨٦ من ٣٢٦)؛ يُطبَّع تلقائياً إلى الصيغة الدولية: `07XXXXXXXXX` ← `+9647XXXXXXXXX` |
| الرصيد | **الرصيد الافتتاحي** | موجب = العميل مدين لنا (يطابق النظام الجديد) — التفصيل §٣ |
| العملة | العملة (IQD/USD) | **اربطه وجوباً مع الرصيد** — رصيد ≠ صفر بلا عملة = فشل الصف (حماية من تفسير دولارات كدنانير) |
| حد الإئتمان | سقف الائتمان | كل قيمه ٠ — ربطه بلا ضرر (يبقى السقف غير مُفعَّل كما هو القرار المؤجَّل) |
| اخر تعامل | آخر تعامل | يُلحق بملاحظات العميل: «آخر تعامل (النظام القديم): YYYY-MM-DD» |
| نشط | نشط | كل القيم TRUE |
| Whatsapp | واتساب | فارغ كلياً — اربطه أو تجاهله، لا فرق |

| عمود الملف | حقل النظام | ملاحظات |
|---|---|---|
| الاسم | الاسم (إلزامي) | |
| الرقم | **الرقم القديم** (legacyCode) | ١٨٥ مملوءاً / ١٨٤ فريداً — **يوجد ازدواج واحد**: الصف الثاني يُعلَّم «مكرّر داخل الملف» برسالة تذكر الرقم؛ راجعه يدوياً وصحّح الملف |
| تليفون 1 | الهاتف | مملوء ٥١٪ (٩٤ من ١٨٦)؛ تطبيع دولي تلقائي |
| الرصيد | **الرصيد الافتتاحي** | **الأقواس المربعة `[27,749,996.]` = سالب** في النظام القديم؛ الافتراض «اعكس الإشارة» — التفصيل §٣ |
| العملة | العملة (IQD/USD) | **صفّان فارغان** في ملفك (١٨٤ من ١٨٦): إن كان رصيدهما ≠ صفر يفشلان برسالة «حدّد العملة» — أكمل العملة في الملف ثم أعد |

### ٢.٣ مراجعة الأصناف (٩٬٤١٥ صفاً — يُستورد)

| عمود الملف | حقل النظام | ملاحظات |
|---|---|---|
| الاسم | اسم المنتج (إلزامي) | ٩٬٣٤١ اسماً فريداً من ٩٬٤١٥ — **الصفوف المتشاركة الاسمَ تُجمَّع متغيّراتٍ لمنتج واحد** (كل صف بكوده الخاص) |
| الكود | الباركود | فريد ١٠٠٪ — ويُشتق منه **SKU تلقائياً** (لا عمود SKU في ملفك ولا حاجة له) |
| الوحدة | اسم الوحدة | ٩ قيم (قطعة/باكيت/سيت/كيس/كارتون/…)؛ القيمة الإنجليزية **"each" تُطبَّع تلقائياً «قطعة»** — واجهة الفواتير والطباعة عربية |
| المجموعة | الفئة | مملوءة ٤٣٪ (١٨٣ فئة) — الفئات **تُنشأ تلقائياً** بأسمائها؛ الفارغة ⇒ منتج بلا فئة (يُصنَّف لاحقاً) |
| سعر الشراء | كلفة الوحدة | |
| سعر الجملة | سعر الجملة | **القيمة ٠ ⇒ لا يُنشأ سعر جملة** لهذا الصنف (لا «جملة صفرية» وهمية) |

> المخزون الافتتاحي يُسجَّل **حركة جرد افتتاحي** (OPENING) على فرعك الحالي داخل نفس معاملة إنشاء
> المنتج — يظهر في حركات المخزون بمرجعه، لا «رقماً مزروعاً» بلا أثر.

### ٢.٤–٢.١١ الملفات غير المستورَدة (للمرجع — لماذا كل عمود ليس وجهة استيراد)

**مراجعة مبيعات (٢٠٤٬٣٤٦ صفاً):** اسم الزبون / هاتف الزبون / عبر / عنوان الزبون / التاريخ / الوقت /
الرقم / الاسم (حساب التعامل) / المستخدم / عدد الأصناف / اجمالي قبل الخصم / اجمالي الخصم / الملاحظات /
الاجمالي / المدفوع / الباقي — **مستندات تاريخية لا تُستورد** (§٥). أعمدة «اسم/هاتف/عنوان الزبون» يمكن
أن تخدم استخراج عملاء جدد **حصراً بعد التنظيف الإلزامي** (§٨) — الملف الخام ممنوع تلقيمه.

**مراجعة مشتريات (٢٬٨٥٦):** التاريخ / الوقت / الرقم / الاسم / عدد الأصناف / اجمالي قبل الخصم /
المستخدم / اجمالي الخصم / الضريبة / الاجمالي / المدفوع / الباقي — لا يُستورد؛ أثره المالي الصافي
موجود في عمود «الرصيد» بملف الموردين.

**مراجعة مرتجع مبيعات (١٬٠٥٥)** و**مرتجع مشتريات (٩١):** نفس بنية المستندات (تاريخ/رقم/اسم/إجماليات) —
لا تُستورد؛ صافي أثرها داخل أرصدة الملفّين الرئيسيين.

**مراجعة استلام نقدية (٢٬٥٠٦)** و**دفع نقدية (٥٤٧):** التاريخ / الوقت / رقم التعامل / الاسم / المبلغ /
العملة / الاجمالي / الملاحظات / الخصم / المستخدم / المرفقات — سندات قبض/دفع تاريخية؛ أثرها الصافي
داخل «الرصيد». (لاحظ: النظام القديم نفسه حوّل USD إلى IQD في عمود «الاجمالي» — استعملنا ذلك للتحقق
من سعر الصرف، §٣.٢.)

**مراجعة الإيرادات (١٬٠٧٤):** التاريخ / الرقم / البيان / المبلغ / عدد البنود / رقم القيد / الضرائب /
الاجمالي — إيرادات يومية مجمَّعة لقسم الطباعة والاستنساخ؛ لا مقابل لها كمستند تاريخي، وهي أساس توصية
«وحدة إيرادات المطبعة اليومية» (§٩).

**مراجعة المصروفات (٣٬٤٨٤):** التاريخ / البيان / العملة / المبلغ / عدد البنود / رقم القيد / المستخدم /
الاجمالي / المرفقات — شاشة المصروفات اليومية في النظام الجديد تتولّى المصروفات من يوم التشغيل؛
التاريخية تبقى في أرشيف النظام القديم.

## ٣. القرارات المالية — الإشارة والعملة (بأمثلة محسوبة من بياناتك)

### ٣.١ صيغ الأرقام في ملفاتك تُفهَم تلقائياً
- فواصل الآلاف والنقطة الزائدة: **«2,988,100.» ← 2988100**.
- الأقواس المربعة = سالب (اصطلاح ملف الموردين): **«[27,749,996.]» ← ‎−27749996**؛ والسالب الصريح «‎-28920682.01» يُقرأ كما هو.
- التقريب لمنزلتين (نصف لأعلى): «2291.678» ← «2291.68». كل الحساب بحساب عشري دقيق — لا تقريبات ثنائية عائمة.

### ٣.٢ العملة وسعر الصرف
- إن احتوى الملف صفوف **USD** يظهر حقل «سعر صرف الدولار» ويصير **إلزامياً**؛ أرصدة USD تُحوَّل ديناراً به ثم تُقرَّب لمنزلتين. النظام الجديد **دينار فقط** — لا عمود عملة دائماً.
- **مثال حقيقي من ملفك «دفع نقدية»:** المبلغ 544.5 USD والإجمالي 789,525 د.ع ⇒ النظام القديم نفسه استعمل سعر **1450**. بنفس السعر: `544.5 × 1450 = 789,525.00 د.ع` — تطابق تام.
- **حماية مدمجة:** إن ربطتَ «الرصيد» وتركت عمود «العملة» غير مربوط، يُمنع التنفيذ بتحذير بارز. السبب بمثال حقيقي: الرصيد «977,899.5» لو كان دولارياً وخُزّن حرفياً ديناراً لضاع فرق ×1450 (الفرق ~١٫٤ مليار د.ع) **بصمت**.
- **رصيد ≠ صفر مع عملة فارغة = فشل الصف** برسالة «حدّد العملة» (في ملف الموردين صفّان كهذان) — لا افتراض صامت للدينار.

### ٣.٣ اتجاه الإشارة — أخطر قرار في الاستيراد كله
| | ملف العملاء | ملف الموردين |
|---|---|---|
| اصطلاح النظام القديم | موجب = العميل مدين **لنا** | الأقواس/السالب = **نحن** ندين للمورد |
| اصطلاح النظام الجديد | موجب = ذمة مدينة لنا (AR) | موجب = ذمة دائنة علينا (AP) |
| **الافتراض في شاشة الاستيراد** | **«كما في الملف»** | **«اعكس الإشارة»** |

**أمثلة محسوبة:**
- عميل برصيد «2,988,100.» (IQD، كما في الملف) ⇒ يُخزَّن **‎+2,988,100.00** = العميل مدين لنا — يظهر في أعمار الذمم المدينة.
- مورد برصيد «[27,749,996.]» ⇒ يُقرأ ‎−27,749,996 ⇒ مع «اعكس الإشارة» يُخزَّن **‎+27,749,996.00** = نحن ندين للمورد — يظهر في الذمم الدائنة.
- مورد برصيد «‎-28920682.01» ⇒ مع العكس يُخزَّن **‎+28,920,682.01**.

**وقاية من القلب الخاطئ:** شاشة الاستيراد تعرض **معاينة حيّة** لصفوف حقيقية من ملفك مفسَّرةً
(«[27,749,996.] ⇐ نحن ندين للمورد ٢٧٬٧٤٩٬٩٩٦ د.ع») تتغيّر فور تبديل الاتجاه — **اقرأها قبل التنفيذ**؛
قلب الإشارة يقلب ذمم الملف كله دفعة واحدة.

## ٤. الرصيد الافتتاحي — ماذا يحدث له بالضبط وكيف يُسوّى

**ما لا يحدث:** لا نُولّد فواتير وهمية ولا قيود مبيعات/مشتريات تاريخية تقابل الرصيد — الرصيد الافتتاحي
**ليس إيراداً ولا مصروفاً**، وتلفيق مستندات له يفسد تقارير المبيعات والأرباح من اليوم الأول (§٥).

**ما يحدث:**
1. يُكتب الرصيد (بعد تحويل العملة وتطبيق الاتجاه) في **الرصيد الجاري** للعميل/المورد عند **الإنشاء فقط** — وفي نفس المعاملة الذرّية.
2. يُرسَّخ بقيد دفتر **مرجعي** من نوع «رصيد افتتاحي» (OPENING): مبلغه = الرصيد الموقَّع، **بلا إيراد ولا كلفة ولا ربح** (أصفار) — أثره الوحيد توثيقي ومحاسبي ضبطي. ملاحظته: «رصيد افتتاحي (استيراد من النظام القديم)».
3. لكل طرف **مفتاح منع ازدواج فريد** على هذا القيد ⇒ مهما أُعيد تشغيل الاستيراد **يستحيل** تكرار الرصيد أو قيده — حماية على مستوى قاعدة البيانات لا البرنامج فقط.
4. أداة **الضبط المالي** (reconcile) تشمل قيود OPENING في معادلتها ⇒ بعد الاستيراد تُظهر **صفر انحراف** — المستورَد لا «يُغرق» تقرير الضبط بانحرافات زائفة.
5. **التحديث لا يلمس الرصيد:** إن كان الطرف موجوداً مسبقاً يُتجاهَل رصيد الملف مع رسالة صف واضحة — تعديل أرصدة الموجودين يكون من سنداته لا من ملف استيراد.

**التسوية لاحقاً — بالسندات لا بتعديل الأرقام:**
- يسدّد العميل ⇒ **سند قبض** (شاشة الذمم/كشف الحساب) يخفض رصيده ويدخل الصندوق ويُقيَّد دفترياً.
- نسدّد المورد ⇒ **دفعة مورد** كذلك.
- ينخفض «الرصيد غير المفوتر» تدريجياً حتى الصفر، وكل ما بعده فواتير حقيقية في النظام الجديد.

**أين يظهر المستورَد في التقارير؟** في **أعمار الذمم** (المدينة AR والدائنة AP) برصيده الجاري — لكن
**خارج الدلاء الزمنية** (٣٠/٦٠/٩٠ يوماً) لأن الدلاء تُبنى من فواتير غير مسوّاة والرصيد الافتتاحي ليس
فاتورة؛ الفرق بين «الرصيد الجاري» و«غير المسدَّد المفوتر» هو **الرصيد غير المفوتر (يشمل الافتتاحي
المستورَد)** ويُعرض عموداً توضيحياً في تقريري الأعمار وسطراً في كشفَي الحساب ضمن هذه الشريحة. متابعة
التحصيل — وهي سبب استيراد الأرصدة أصلاً — تتم من هناك.

## ٥. لماذا لا تُستورد المستندات التاريخية (مبيعات/مشتريات/مرتجعات/نقدية/مصروفات)؟

1. **خطر التكرار المالي:** كل مستند مستورَد سيحاول كتابة قيود دفتر وحركات مخزون وأثر ذمم — فوق أرصدة
   افتتاحية **تحمل صافي ذلك التاريخ أصلاً** ⇒ ازدواج حتمي للأموال.
2. **يكسر مناعة النظام:** آلية منع التكرار في النظام الجديد مبنية على مفاتيح مصدر فريدة لعملياته هو؛
   حقن ٢٠٤ ألف مستند خارجي يربك هذه الضمانة الذرّية.
3. **بيانات غير قابلة للإسقاط أصلاً:** مستندات النظام القديم بلا تفاصيل أصناف مرتبطة بكتالوج النظام
   الجديد (أسطر الفواتير غير موجودة في التصدير)، وهواتف تصل ١٢٨ محرفاً، وأسماء حسابات وسيطة
   («تعامل نقدي»، «قسم الطلبات») لا تقابل عملاء حقيقيين.
4. **الأرصدة الافتتاحية تغني عنها:** صافي كل ذلك التاريخ = رقم واحد لكل طرف، وهو بالضبط ما استوردناه
   بقيد OPENING مرجعي. التاريخ التفصيلي يبقى في **أرشيف النظام القديم للمراجعة** عند الحاجة.

## ٦. حالات خاصة في ملفاتك — مقيسة ومعالجَتها منصوصة

- **الهواتف المشتركة (ليست خطأ):** في ملف العملاء ١٨٦ هاتفاً مملوءاً منها ١٨٠ فريداً، وفي الموردين
  ٩٤/٨٨ — أي **٦ صفوف في كل ملف تتشارك هواتف** مشروعاً (عائلة واحدة/محل واحد برقم واحد). لذلك
  **الهاتف وحده ليس مفتاح كشف تكرار**: المفتاح هو **الرقم القديم** إن وُجد، وإلا (الهاتف+الاسم) معاً،
  وإلا الاسم — فلا تُرمى هذه الصفوف «مكرّرات» ولا تسقط أرصدتها بصمت.
- **ازدواج الرقم القديم الوحيد في ملف الموردين** (١٨٥ مملوءاً/١٨٤ فريداً): الصف الثاني يُعلَّم «مكرّر
  داخل الملف» برسالة تذكر الرقم المزدوج — صحّحه في الملف قبل التنفيذ.
- **المخزون السالب** (شائع في الأنظمة القديمة — بيع على المكشوف): **يُقصّ إلى صفر** ويُستورد الصف مع
  تحذير «رصيد سالب في النظام القديم — استُورد صفراً، سوِّه من شاشة الجرد». بعد الاستيراد افتح حركات
  المخزون وسوِّ هذه الأصناف بجرد فعلي.
- **«4.» بنقطة زائدة** في أعمدة الأعداد: تُقرأ ٤ بصمت؛ أما الكسري الحقيقي (4.5 قطعة) فيفشل الصف —
  المخزون بالوحدة الصحيحة.
- **"each" ← «قطعة»:** طُبّعت تلقائياً — لا وحدات لاتينية في فواتيرك وطباعتك.
- **حد الإئتمان في ملف الموردين:** يُتجاهَل عمداً (كل قيمه صفر ولا مفهوم «سقف ائتمان مورد» في النظام).
- **أسماء عملاء تبدو باركودات** («8437020176569»): تُستورد كما هي حفاظاً على أرصدتها — نظّف الأسماء
  لاحقاً من شاشة تعديل العميل (الرقم القديم يبقى رابطك للنظام القديم).

## ٧. خطوات الاستيراد عملياً — شاشة بشاشة

> قبل البدء: خذ نسخة احتياطية (`pnpm db:backup`)، وتأكّد أنك بحساب مدير وعلى الفرع الصحيح
> (المخزون الافتتاحي يُسجَّل على فرعك الحالي).

### ٧.١ الأصناف (شاشة «المنتجات» ← زر «استيراد Excel»)
1. اختر ملف «مراجعة الأصناف». الربط تلقائي: الاسم/الكود/الوحدة/المجموعة/سعر الشراء/سعر الجملة/سعر البيع/الرصيد. اترك «اجمالي التكلفة» **— تجاهل —**.
2. راجِع المعاينة: ستظهر تحذيرات قصّ المخزون السالب (إن وُجد) وأخطاء الصفوف (إن وُجدت). افتراضياً **لا يُكتب شيء حتى تخلو كل الصفوف من الأخطاء** — صدِّر «سجلّ الأخطاء»، صحِّح ملفك، وأعد.
3. نفّذ. الملف يُرسَل **~١٠ دفعات** (≤١٠٠٠ صف، دون فصم منتج متعدد الصفوف عبر دفعتين) مع شريط تقدّم. إن توقّفت دفعة تظهر رسالة تبيّن ما كُتب وما لم يُكتب — **إعادة التشغيل آمنة** (الموجود يُتخطّى).
4. تحقّق: عدد المنتجات، عيّنة فئات أُنشئت، عيّنة أصناف برصيدها في حركات المخزون (مرجع «رصيد افتتاحي»).

### ٧.٢ الموردون (شاشة «الموردون» ← «استيراد Excel»)
1. اختر ملف «مراجعة الموردين». الربط تلقائي؛ «حد الإئتمان» يبقى غير مربوط (مقصود).
2. **خيارات الاستيراد:**
   - **اتجاه الرصيد: «اعكس الإشارة» (الافتراض — لا تغيّره)**. اقرأ المعاينة الحيّة: «[27,749,996.] ⇐ نحن ندين للمورد …».
   - **سعر صرف الدولار:** يظهر لوجود صفوف USD — أدخل سعر يوم الاستيراد (القديم استعمل 1450 — §٣.٢).
3. عالج المتوقَّع: ازدواج الرقم القديم الواحد + صفّا العملة الفارغان (إن كان رصيدهما ≠ صفر) — صحّح الملف وأعد، أو فعّل «تجاوز الصفوف الفاشلة» لتستورد الصالحَ الآن وتعالج البقية يدوياً.
4. تحقّق: عمود «الرقم القديم» في الشاشة، أرصدة عيّنة (٢-٣ موردين كبار) مقابل النظام القديم، وتقرير أعمار الذمم الدائنة.

### ٧.٣ العملاء (شاشة «العملاء» ← «استيراد Excel»)
1. اختر ملف «مراجعة العملاء». الربط تلقائي.
2. **اتجاه الرصيد: «كما في الملف» (الافتراض)** — موجب = العميل مدين لنا. راجع المعاينة الحيّة. أدخل سعر الصرف إن ظهر (صفوف USD).
3. نفّذ وتحقّق: أرصدة عيّنة عملاء كبار، وتقرير أعمار الذمم المدينة، وكشف حساب عميل مستورَد (سيُظهر الرصيد غير المفوتر).
4. من هنا فصاعداً: التحصيل بسندات القبض — الرصيد الافتتاحي ينخفض بها تدريجياً (§٤).

### ٧.٤ بعد الاستيراد الثلاثي
- شغّل **الضبط المالي** وتأكّد من **صفر انحراف**.
- خذ نسخة احتياطية ثانية (لقطة «ما بعد الاستيراد»).
- **أوقف الإدخال في النظام القديم** — نظامان حيّان يتباعدان من أول فاتورة، وأبقِه أرشيفاً للقراءة.

## ٨. استخراج عملاء من ملف المبيعات — ممكن، لكن بعد تنظيف إلزامي

ملف المبيعات الخام (٢٠٤ ألف صف) **ممنوع تلقيمه لشاشة الاستيراد**: أسماء الزبائن مكرّرة بكثافة
(~٧ آلاف فريد من ~١٥٫٦ ألف مملوء)، وخانة الهاتف تصل **١٢٨ محرفاً** (هواتف متعددة محشورة) فترفضها
حدود التحقق، والنتيجة فشل شامل فوضوي. إن أردت ضمّ زبائن الطلبات/التوصيل:

1. **جمّع بالاسم** (صف واحد لكل زبون) في Excel — Pivot أو إزالة تكرار.
2. **فصّل الهواتف المتعددة**: هاتف واحد في «هاتف الزبون» (≤ ٢٠ محرفاً)، والبقية في أعمدة إضافية أو احذفها.
3. أبقِ أعمدة: اسم الزبون، هاتف الزبون، عنوان الزبون (ترويساتها معروفة لشاشة استيراد العملاء).
4. استورد الملف المنظَّف من شاشة العملاء — **بلا عمود رصيد** (لا أرصدة في ملف المبيعات؛ الذمم جاءت من ملف العملاء).
5. لاحظ أن المطابقة بالهاتف/الاسم ستتخطّى الموجودين تلقائياً.

## ٩. توصيات مستقبلية (خارج هذه الشريحة — للقرار لاحقاً)

1. **وحدة إيرادات المطبعة اليومية:** ملف «الإيرادات» (١٬٠٧٤ قيداً بنمط «قسم الطباعة والاستنساخ»
   اليومي) يكشف عادة تشغيلية راسخة: إقفال إيراد القسم يومياً برقم مجمَّع. تستحق شاشة مكرّسة
   (إيراد يومي لكل قسم ← قيد دفتر + صندوق) بدل إدخالها «مبيعات» صورية.
2. **حقل قناة التواصل «عبر»:** عمود «عبر» في المبيعات (واتساب/انستغرام/مباشر… ٤٨٠ قيمة بصيغ مختلفة)
   مفيد للتسويق — يستحق حقل enum منضبطاً في بطاقة العميل مستقبلاً.
3. **الرقم القديم في الشاشات (مُنفَّذ):** عمود «الرقم القديم» يظهر في شاشتي العملاء والموردين متى
   وُجدت قيم (ويُصدَّر في Excel)، وخانة البحث فيهما تلتقطه — ابحث برقم النظام القديم مباشرة.
4. **تنظيف ما بعد الاستيراد:** أسماء العملاء الرقمية، توحيد أسماء الفئات المتشابهة إملائياً
   (في ملفك «قسم الطباعة والاستنساخ» بـ ٦+ هجاءات)، وجرد فعلي للأصناف التي قُصّ رصيدها السالب.
5. **الأرشيف:** احفظ ملفات التصدير الأحد عشر + نسخة قاعدة النظام القديم في مكان للقراءة فقط —
   هي مرجعك التاريخي الوحيد بعد إيقاف النظام القديم.
