# دليل التشغيل في ظروف العراق (كهرباء/إنترنت متذبذبان)

> مرجع عملي للمالك/المشغّل. يغطّي: التشغيل التلقائي، الصمود أمام انقطاع الكهرباء، استقلالية الإنترنت،
> النسخ الاحتياطي والاستعادة، تصفير النظام، والفرعين. خيارات لاحقة موصوفة في النهاية.

## ١. التشغيل التلقائي عند الإقلاع (مُنفَّذ)
بعد عودة الكهرباء يجب أن يعود النظام **دون تدخّل بشري**. التهيئة لمرة واحدة:

```powershell
# 1) ابنِ النسخة الإنتاجية
pnpm build

# 2) شغّل الخادم عبر PM2 واحفظ القائمة
pm2 start ecosystem.config.cjs
pm2 save

# 3) فعّل Docker Desktop ليبدأ عند تسجيل الدخول:
#    Docker Desktop → Settings → General → "Start Docker Desktop when you sign in"

# 4) سجّل مهام الإقلاع + الصيانة (بصلاحيات مدير):
powershell -ExecutionPolicy Bypass -File scripts\install-autostart.ps1
```
يسجّل ذلك ثلاث مهام: **Boot** (تشغيل حاوية MySQL ثم `pm2 resurrect` عند الدخول)، **Docker Watchdog** (كل ساعة)، **Daily Backup** (2ص). تحقّق: `Get-ScheduledTask -TaskName 'AlRoya ERP*'`.

## ٢. الصمود أمام انقطاع الكهرباء
**ما يصمد تلقائياً:**
- **سلامة البيانات:** MySQL InnoDB ذرّي وقابل للاسترداد بعد قطع مفاجئ — أي عملية أُكملت (`COMMIT`) محفوظة، وأي عملية في منتصفها تُلغى كاملةً (لا فواتير نصف مكتملة). كل عمليات الأعمال داخل `withTx`.
- **العودة التلقائية:** PM2 يعيد الخادم، وحاوية MySQL (`restart: unless-stopped`) + مهمة Boot تعيدان القاعدة.
- **إغلاق رشيق:** عند SIGTERM/SIGINT يُغلق الخادم اتصالاته والقاعدة برفق (`kill_timeout` في PM2).

**🔴 موصى به بشدّة قبل البيانات الحقيقية — UPS:**
انقطاع الكهرباء المتكرّر المفاجئ هو أكبر خطر على القاعدة. ركّب **UPS** (بطارية احتياطية) لجهاز الخادم + الراوتر:
- يمنح وقتاً لإطفاء رشيق بدل القطع الفجائي (يقلّل خطر تلف ملفات القاعدة).
- اضبط برنامج الـUPS لإصدار `pm2 stop erp-server` ثم إطفاء Windows عند هبوط البطارية.
- 15–30 دقيقة بطارية تكفي لتجاوز الانقطاعات القصيرة دون إطفاء أصلاً.

**بعد انقطاع غير رشيق:** افحص الصحّة `Invoke-WebRequest http://localhost:3000/healthz`، وراجع `logs/erp-error.log`. عند أي شكّ في سلامة القاعدة، استعد آخر نسخة سليمة (§٤).

## ٣. استقلالية الإنترنت (مُنفَّذ)
النظام **محلّي بالكامل**: المتصفّح يكلّم خادماً على `localhost`، ولا يعتمد على أي خدمة خارجية للعمل.
- **الخطوط مستضافة محلياً** (`@fontsource/cairo`) — أُزيل اعتماد Google Fonts CDN ⇒ يعمل النظام بمظهره كاملاً **بلا إنترنت إطلاقاً**.
- لا استدعاءات API خارجية في منطق الأعمال. Sentry معطّل افتراضياً.
- انقطاع الإنترنت لا يؤثّر على البيع/الطباعة/الجرد ما دام جهاز الخادم وMySQL يعملان.
- اعتماد الإنترنت الوحيد: تثبيت الحزم (`pnpm install`) والتحديثات — تُجرى عند توفّره فقط.

## ٤. النسخ الاحتياطي والاستعادة
**النسخ (تلقائي + يدوي):**
```powershell
pnpm db:backup           # نسخة الآن (mysqldump ذرّي --single-transaction) + تدوير 7/4/3 + نسخة خارجية
```
- مجدول يومياً 2ص عبر مهمة Daily Backup. النسخ في `backups/` (مُستثنى من git).
- اضبط `BACKUP_OFFSITE_DIR` في `.env` لنسخة خارجية (OneDrive/USB) — **حماية من فقد الجهاز**.

**الاستعادة (CLI آمن):**
```powershell
pnpm db:restore backups\erp-<التاريخ>.sql --confirm RESTORE
```
يأخذ نسخة أمان للحالة الراهنة أولاً، يتحقّق من الملف، ثم يكتب فوق القاعدة. (للخدمة الأصلية: أضِف `--native`.)

**اختبار الاستعادة دورياً:** استعد آخر نسخة إلى قاعدة اختبار كل ربع سنة للتأكّد أنها سليمة فعلاً (نسخة موجودة ≠ نسخة قابلة للاستعادة).

## ٥. تصفير النظام (البدء من جديد)
لمسح كل بيانات التجربة والبدء بنظام فارغ (يُبقي المستخدمين + الفرعين فقط):
```powershell
pnpm db:reset --confirm RESET           # نسخة احتياطية تلقائية ثم تصفير كامل
pnpm db:reset --confirm RESET --seed    # تصفير ثم إعادة عيّنات (admin/فروع/منتجات عيّنة)
```
عملية لا رجعة فيها (لذا CLI فقط + رمز تأكيد + نسخة احتياطية إجبارية أولاً). تبدأ أرقام الفواتير وكل العدّادات من جديد.

## ٦. الطباعة الصامتة
طباعة فواتير الكاشير مباشرةً دون حوار المتصفّح عبر **جسر الخادم** — اضبط في `.env`:
```
PRINT_TARGET=tcp://192.168.1.50:9100   # طابعة حرارية شبكية RAW (الأوثق، بلا تعريفات)
# أو: PRINT_TARGET=share://POS80        # طابعة USB مشتركة باسم RAW على Windows
```
إن تُرك فارغاً يتراجع النظام تلقائياً إلى WebUSB ثم حوار المتصفّح. زر «🖨️🌐» في شاشة الكاشير يطبع تذكرة اختبار.

## ٧. الفرعان
- **قاعدة مركزية واحدة** للفرعين (الموصى به، مُطبَّق): الفرع الثاني يتصل بقاعدة الفرع الرئيسي عبر الشبكة المحلية/VPN. لا مزامنة معقّدة، والتحويلات بين الفروع ذرّية بقفل ثنائي.
- متطلّب: رابط شبكي مستقرّ بين الفرعين (VPN/خط مخصّص). عند انقطاعه يتوقّف الفرع الثاني عن البيع (انظر «وضع عدم الاتصال» أدناه).

## خيارات لاحقة (موصوفة — تُنفَّذ عند الحاجة)
- **ترحيل MySQL إلى خدمة Windows أصلية** (بدل Docker الهشّ على Windows — علّة «Inference manager»): أصلب ضد الانقطاع المتكرّر لمتجر واحد. الخطوات: `pnpm db:backup` → ثبّت MySQL 8 كخدمة Windows (تبدأ تلقائياً) → أنشئ القاعدة → `pnpm db:restore <نسخة> --native` → حدّث `DATABASE_URL` و`DB_NATIVE=1`. عندها لا حاجة لـDocker/الحارس.
- **وضع عدم الاتصال لفرع المبيعات** (جهد معماري كبير): تخزين محلي + طابور مزامنة كي يستمر الفرع الثاني بالبيع رغم انقطاع الشبكة مع الفرع الرئيسي، ثم يزامن عند العودة. نقطة الانطلاق: PWA مهيّأة جزئياً (`vite-plugin-pwa`).
- **خادم احتياطي/تكرار MySQL** عند التوسّع لـ3+ فروع.
