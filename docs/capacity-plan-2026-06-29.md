## ٢. الجذور التي عُولِجت بالأرقام

### F1 — علّة فهرس حيّة (0030)
`idx_receipt_bucket_status` أُنشئ في 0013 على عمود `bucketId`، حذفه 0017 (إسقاط دلاء النقد) فأسقطه MySQL تلقائياً، ولم يُعَد على `cashBucket`. مُتحقَّق على قاعدة طازجة معزولة بـ`INFORMATION_SCHEMA.STATISTICS`.

### F2 — فهارس مركّبة ناقصة (0031)
١٧ فهرساً على الجداول العالية الحجم (الفواتير/الدفتر/الحركات/الإيصالات/المخزون/التدقيق...) لمنع full scan في التقارير عند الملايين.

### F3 — فلاتر تاريخ غير قابلة للفهرسة (S2 + 0032)
`DATE(invoiceDate)`/`DATEDIFF(NOW(), invoiceDate)` تُبطل الفهرس وتفرض مسح كل الجدول. اُستبدلت بنطاقات قابلة للفهرسة `invoiceDate >= from AND < nextDay(to)`؛ التكافؤ مُثبَت (COUNT/SUM متطابقان حرفياً) + فهارس مُغطّية بترتيب «التاريخ ثم الحالة».

### F7 — ثغرة الوقاية (db:verify)
كان يفحص الأعمدة/الجداول فقط (لا الفهارس) ⇒ فشل 0013 الصامت بقي غير مرئي حتى التدقيق. الآن يؤكّد ١٣ فهرساً حرجاً عبر `INFORMATION_SCHEMA.STATISTICS` في كل `prod:deploy`، مُتحقَّق بالاتجاهين.

### علّة صحّة في POS (S5)
`customers.list.find` على قائمة مقطوعة عند ٥٠٠ ⇒ تسعير RETAIL خاطئ + فقدان الرصيد لأي عميل خارجها. fallback `customers.get` لا يضيف وميض تسعير (قراءة فورية من القائمة للشائع + جلب فقط عند عدم الوجود).

## ٣. اكتشافات مُقاسة لا تظهر إلا تحت الحجم

1. **`DATE(col)` يمسح كل الجدول؛** نطاق قابل للفهرسة يمسح نافذة التاريخ فقط (مستقلّ عن حجم الجدول).
2. **S2 وحده بلا فهرس مُغطٍّ يَنكُص** (٤٨٦٦مس مقابل ٢٨٥٥) بسبب بحث الجدول العشوائي ⇒ الكود + الفهرس المُغطّي حزمة واحدة.
3. **ترتيب الفهرس مهمّ حسب نوع المُسند:** للنفي `NOT IN` (تقارير المبيعات): `(date,status)` أسرع. للشمول `IN` (AR aging): `(status,date)` أسرع بـ٥×. هذا حسم فرضية «status-first يكسر البادئة» — صحيحة للنفي فقط.
4. **كلفة الكتابة:** بذر ٥٠٠ألف فاتورة تباطأ ٧٤٥٦←١٠٧٨/ث مع نموّ الفهارس ⇒ احذر الإفراط (مرحلة 0033 حذفت بادئة مكرّرة).

## ٤. المتبقّي (بترتيب الأولوية)

### عالية — مرشّحات للموجات التالية
- **S3 — keyset pagination** لمسارات OFFSET (sale/purchase/inventory/expense routers). OFFSET يتدهور خطّياً مع عمق الصفحة عند الملايين. النمط: `WHERE id < :cursor ORDER BY id DESC LIMIT n + nextCursor`.
- **S4 — إزالة COUNT لكل صفحة** (sale.listSummary، auditRouter، shiftRouter، expenseService) — استبدال بـ`hasMore` (جلب limit+1) لتفادي مسحين كاملين لكل تحميل صفحة.
- **CustomerPicker → smartSearch**: المنتقي المنسدل ما زال يحمّل ٥٠٠ عميل عند الإقلاع — تحويله لبحث `smartSearch` يُلغي ذلك.
- **حارس انحدار** `server/services/__tests__/perf.explain.test.ts` يبذر بيانات ممثّلة ويؤكّد لكل استعلام: `access_type != ALL`، `key` = الفهرس المتوقّع، `rows_examined <= budget` — فيفشل أي تغيير مستقبلي يُعيد `DATE(col)` أو يُسقط فهرساً.

### متوسطة (تحتاج مراجعة عدائية أو نافذة هدوء)
- **D1 — N+1 + نافذة الأقفال في `saleService.ts`** (لكل سطر فاتورة استعلام `productVariants` تحت قفل المعاملة) — يتطلّب مراجعة مالية/ذرّية لأنّه مسار المال.
- **D2 — FULLTEXT + عمود تطبيع عربي مُخزَّن** بدل `LIKE '%..%'` عبر ٩ جداول و~٣٠ `REPLACE()` وقت الاستعلام (`globalSearchService.ts`، `catalogService.ts`). يتطلّب نافذة بناء فهرس + تعديل خدمات الكتابة.

## ٥. الأدوات الجاهزة للاستعمال

- `pnpm seed:bulk` — مولّد بيانات ضخم بحارس أمان يرفض `erp-mysql-prod`/3306/erp/erp_test. `scripts/bench/seedBulk.mjs`.
- `pnpm bench` — راكض قياس دافئ متعدّد التشغيلات (min/median/max ms) + EXPLAIN. `scripts/bench/runBench.mjs`.
- `pnpm db:verify` — يُتحقّق من snapshot + ١٣ فهرساً حرجاً (يلتقط أي علّة فهرس صامتة).
- بيئة القياس المعزولة الموثَّقة في الذاكرة: [[local-db-containers]].

## ٦. النشر للإنتاج

- موجات ١-٤ كلها في `origin/main` وجاهزة لـ`pnpm prod:deploy` (نسخة احتياطية → db:migrate:safe → db:verify → build → pm2 reload).
- كل الهجرات (0030-0033) **رخيصة جداً الآن** (MySQL 8 InnoDB، DDL ثانوية INPLACE/LOCK=NONE) قبل تضخّم البيانات. أي تأجيل يضاعف كلفة التطبيق.
- النشر **بأمر المالك** فقط (كل النشاطات الإنتاجية).
